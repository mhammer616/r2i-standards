UX and Front End Standards Guide

This guide is meant to be a repository of best practices and standard approaches to designing and developing common elements used on most of our websites. This guide is a work in progress and will evolve over time as we define and refine our best practices and standards.

Basic Guidlines

Every site or landing page that we build probably has at least one Form somewhere on them. Forms can be time consuming to build, style, and make functional, this section will help define some basic resets, and supply some code to help make this easier.

Forms, like the website in general, should follow a grid layout with rows of fields that cover any number of columns.

Forms are made up of various ways for the user to supply information back to the client. The basics of form elements are listed below, along with some custom implementations.

Labels

A label, even if hidden, has to be supplied for usability for every form element.

The label tag must be properly hooked up using the for attribute to associate it with the id of the input.

Inputs

Basic Text Inputs

Basic text inputs can be styled very simply by using CSS to manipulate the look and feel, we can easily control the following types: text, password, number, email, url, search, and tel.

By using the proper type for the input, accessibility is increased by changing the keyboards on mobile and tablet devices, along with allowing basic validation of the elements to be performed.

Use Bootsrap's form element wraps and classes to build out each form element.

Date Input

When using the input type of date, we can see that each browser processes it somewhat differently by outputting their own datepicker, which can't be styled yet. For this we want to use a third party plugin to allow for a more custom view and selection of dates.

The Bootstrap Datepicker by eternicode is the plugin we have chosen to be our standard use. The styling of the datepicker gives us the desired flexibility and usability.

There is even an online sandbox that will write out the code needed to supply the desired end result. Use this to define what options are necessary to create the Datepicker you want to show on the website.

Download the latest version of the plugin on GitHub, or use the version we are using here.

Checkboxes

It's common in our designs that checkboxes are styled. This can be a relatively easy process if the code is setup and written properly. It does however require the use of a different way than the standard approach to setting up checkboxes inputs. But, this approach will require no JavaScript, and allow the checkboxes to function properly.

Radio Buttons

It's common in our designs that checkboxes are styled. This can be a relatively easy process if the code is setup and written properly. It does however require the use of a different way than the standard approach to setting up checkboxes inputs. But, this approach will require no JavaScript, and allow the checkboxes to function properly.

Selects

Selects are often styled in our designs, and they are one of the hardest things to style in HTML and CSS. So we always rely on a plugin to do this. Over the years we've had many different select plugins offering a variety of support, most of which was for styling, however there's only been one that gave us the ability to style in the way we needed as well as the functionality, usability, and accessibility we needed.

Use Chosen by HarvestHQ for all selects to make styling easier. The capabilities this tool gives us are worth doing the custom implementation for, even if you just want to use it with basic styles. The other added benefit, it will look just like the traditional select on mobile devices, still giving us the default functionality people expect when on the go.

Download the latest version of the plugin on GitHub, or use the version we are using here.

Textareas

Textareas are used to show multiple lines for a response, commonly used for Comments, Concerns, or a freeform box.

Placeholder Forms

Placeholder forms have become very popular, but sometimes aren't as usable as one would like to think. The thought is you can fit forms into smaller areas using the placeholder as a label, however that is not the proper use of the placeholder, the placeholder is meant to supply supplemental information to the user on how to utilize that field. Like a telephone or date format, zip code format, or something like that. This also decreases the usability for accessibility use cases, screen readers and audio clues become lost when the label isn't present. Older browsers, like IE9, don't support the use of placeholder, and while plugins exist, they can cause other issues with autocomplete features.

So when developing a form that looks like it makes use of placeholders, use the label and overlay it on top of the input, then have it either move above of the form, or have the text to the right. See below for both examples.

Overview

We have three different accepted best practices for Carousels. If Bootstrap Carousel fits your needs, it should be used first, but sometimes, we have a need to show multiple slides at once which Bootstrap was not built for, step in Owl Carousel and jCarousel.

Before diving into the code, decide which of the three is right for your website. Ask the following:

  • Which one most looks like the design?
  • Do I need to have multiple slides visible at once?
  • What controls do I need to include?
  • Do I need a play/pause button?
  • What transitions do I need to switch between slides?

Overview

We try not to have too many standards and best practices, and try to give people the freedom to code how they are used to. However, we do have a few things that we have defined so that we do write our code in a similar fashion, so that we can pick up projects working as a team and hop on and off as needed.

By following a few simple rules and not getting to high level or in the weeds we can hand off projects with little to no documentation to other front end developers on the team.

Our Basic Guidelines

  1. Style using classes and use id's for JavaScript.

    So this one started because DNN would change id's and has since just become we have adopted as our standard. Please avoid styling using ID's in your CSS or Less files, just use classes. Most of the CMS's we utilize will manipulate some id's on the page, or the id's are being dynamically controlled, so they are unreliable to use in CSS or Less.

  2. Name your classes to what they are used for not what they look like.

    The color, size, or shape of something may change over the course of project, so name something for it's functional use, not the way it looks.

    If you have similar functional use cases, like three different carousels, start them all similiarly like "carousel-" specific use. This will keep things organized, and allow for specific styles to be set for each use.

  3. Style using classes and use id's for JavaScript.

    So this one started because DNN would change id's and has since just become we have adopted as our standard. Please avoid styling using ID's in your CSS or Less files, just use classes. Most of the CMS's we utilize will manipulate some id's on the page, or the id's are being dynamically controlled, so they are unreliable to use in CSS or Less.

  4. Spaces over tabs.

    For coding consistency with all our development team and source control, we ask that you use 4 spaces instead of 1 tab. Most of the IDE's we use offer the ability to set this in the settings so you can use the tab and make it equal 4 spaces.

  5. CSS and Less formatting

    Again for ease of use with Source Control, we ask that you write each definitions of CSS or Less on a new line. Selector's can be done on a single line.

Progressive Enhancement vs. Graceful Degradation

As a team, and as a company, we try to take the approach of progressively enhancing our websites, opposed to gracefully degrading the experience. While this may seem like it's saying the same thing in different ways, they are fundamentally different.

First, let's look at how we define approach:

Progressive Enhancement: start by establishing our basic level, this is done by identifying what our base requirement of support is for the website, then we can build our base element to this specification. After that we enhance the code by adding support for new features, like transitions, transforms, transparent background elements, hover effects, etc. We should only be supporting features that are supported by each browser, accepting that there may be slight differences between browser experiences.

Graceful Degradation: start by building out the final result with all the bells and whistles. Then use plugins, polyfills, and hacks to make older browsers work the same.

We utilize modernizr.js, to test whether a feature that we want to use is supported or not supported, then write our CSS accordingly. Modernizr.js will add a class to the html tag identifying if the desired feature you tested for is or isn't available for use. Then you can make different definitions based on that feature.

We do accept that each browser may vary in appearance slightly, however, the majority of the look and feel should be similar. Everything does need to function correctly, so a form has to submit or a wizard has to step you through each step of the process, but it doesn't have to animate between each step the same in every browser.

While we try to be strict on our approach, there are some exceptions, especially where a couple small lines of JavaScript can make something work across all browsers.

Here are some things we want do, and how we want to do them:

Animations: all animations should be done using CSS wherever possible, you can trigger the animation by adding a class to the element, or changing an attribute with JavaScript or jQuery, but the styles that are altered should be managed within CSS.

Transitions: same rules as animations

SVG: using SVG for icons and images as much as possible, this may require the use of a third party tool like SnapSVG to do more sophisticated drawing and animations.

Text Shadows and Box Shadows: using CSS only to achieve this, if they don't exist in older browsers that's an acceptable difference, it's not worth the additional markup or code and use of images to create such small effects.

Rounded Corners: using CSS border-radius for this feature, if an older browser doesn't support it, they will just get squared off corners, again the extra weight of markup and images is not worth it to create rounded corners.

Video: using HTML 5 video for backgrounds or even self-hosting videos is a great approach, however, get a Flash player to fallback to using the HTML tags properly to work on older browsers. For the backgrounds, just load an image instead of the video.

Folder Structure

Our ideal folder structure, which is what we want used when developing websites without CMS's is the following:

  • assets
    • global
      • JavaScript files
        • modernizr.js
        • jquery.js
        • bootstrap.js
      • Less files
        • variables.less
        • mixins.less
        • fonts.less
      • Image files
    • template-folders
      • JavaScript files
      • Less file that imports available components on that page
        • @import url(../global/fonts.less);
        • @import url(../global/variables.less);
        • @import url(../global/mixins.less);
        • @import url(../{component-folder}/{less-file}.less);
      • HTML file (example markup of template, not necessary)
      • Image files
    • component-folders
      • JavaScript files
      • Less files
      • HTML file (example markup of component, not necessary)
      • Image files

Template naming convention should be template-{type of template}.

Component naming convention should be {component use}-{specific component type}.

For example: Main Navigation should be nav-main, and interior navigation should be nav-interior or nav-sub.

Another example: blog components list-blog, detail-blog, categories-blog, related-articles-blog.

Modernizr

Modernizr.js is an online utility that allows us to identify and detect if a browser supports a speecial feature. Outlined in the Progressive Enhancement vs. Graceful Degradation we said why we want to detect features over browsers and supply fallbacks for styling or features that may not be supported. We always want to try to enhance the experience over degrading, so keep that in mind when using detection.

When using this tool, you can create a version custom built for your website, or you can load all the detections. jQuery

We know there are choices when picking a library or framework to work with, we've chosen jQuery as our standard. We made this decision because all the CMS's we build in use jQuery.

Some quick guidelines, and best practices we use:

  1. Use .on() to bind to events.

    Instead of using .click(), .mouseenter(), .mouseout() to bind a function use the .on() and .off() to bind and unbind functions to those events.

    2. Except for the initial $(document).ready() call.

  2. Use plugins when necessary.

    Feel free to utilize additional plugins as you may see fit, but try not to use one if it's a simple 5-10 line function.

  3. Don't re-invent the wheel.

    Don't try to write something better than the plugins we've found, it opens us up for additional rounds of QA, vet any new plugin that you want to use through a senior team member so they can give feedback.

  4. Be Consistent.

    Be consistent with your use of punctuation and $ vs jQuery. We don't care which one you use " or ', $ or jQuery, , vs ; and a new line for variables, just pick one way and be consistent on that project. Don't switch back and forth. If we notice a problem with this one, we may implement a standard around it, but for now, just be consistent.

Bootstrap

Again, we know there are other plugins and libraries of code that do something similar to Bootstrap, whether it's jQuery Tools, 960Grid.js, jQuery UI, and countless plugins that give you all the pieces that Bootstrap can, we've chosen to use Bootstrap. It gets us the biggest bang for the buck.

Don't over think it and try to force a square peg in a round hole, if Bootstrap doesn't work or have the component you need to build out, add to it. Find a plugin to supplement on to the Bootstrap and jQuery library we have.

But, don't find a new plugin for something that Bootstrap already has in it's aresenal.
