Basic Coding Standards

We try not to have too many standards and best practices, and try to give people the freedom to code however they are accustomed. However, we do have a few things defined so that when we write our code, it is in a similar fashion. This allows us to work as a team to pick up projects and hop on and off as needed.

By following a few simple rules and not getting too high-level or in the weeds, we can hand off projects with little to no documentation to other front end developers on the team.

Responsive Design

All our websites, emails, and landing pages should be responsive.

If we are doing the designs for the website, you will receive a home page design for both mobile and desktop versions of the website. Then, you wil receive either interior designs for each unique page, or a component library design to build the rest of the website. If budget and timeline allow you might receive additional responsive concepts for complex components, or if you really need something ask your project manager to request the designer to design something out. We have decided on this workflow to do a couple things, a) it saves budget and time; b) it gives us flexibility in how we write the markup.

Mobile First

All sites, unless it's not responsive, should be built with the mobile device in mind first, then enhancing the site as the browser gets larger (more on enhancing the site later). This means, your mobile code should be accessed without the use of a media query.

Here's a few reasons why we're going mobile first:

  • Phones Processor
    The processor on a phone is not as strong as the processor on a computer, connection speeds aren't as good, and therefore you don't want to load things that aren't necessary.
  • Simpler Base
    Generally the mobile device has simpler styles, but are still used to set a base for the others, this creates more streamlined code.
  • Fewer Overrides
    You'll probably have fewer things you'll need to override starting with the mobile device, you won't have to take out the annoying hover effects that only work on larger browsers.

This approach also allows you to code media queries using min-width only and go in a single direction, making the code much easier to understand.

Configure the Viewport

By placing this meta tag: <meta name="viewport" content="width=device-width, initial-scale=1">; at the top of your page you can make the page work on mobile and desktop.

Additonal settings can be applied:

Property Description
width The width of the virtual viewport of the device.
device-width The physical width of the device's screen.
height The height of the "virtual viewport" of the device.
device-height The physical height of the device's screen.
initial-scale The initial zoom when visiting the page. 1.0 does not zoom.
minimum-scale The minimum amount the visitor can zoom on the page. 1.0 does not zoom.
maximum-scale The maximum amount the visitor can zoom on the page. 1.0 does not zoom.
user-scalable Allows the device to zoom in and out. Values are yes or no.

Generally, unless you are building an app, or absolutely don't want the user to scale, don't prevent scaling, as it is potentially opening a door for accessibility issues.

Accessibility Guidelines

All our websites, emails, and landing pages should be accessible.

At a basic level we should code our websites to follow the Web Content Accessibility Guidelines 2.0 (WCAG) which covers a range of recommendations that will make content accessible to people with disabilities, including blindness and low vision, deafness and hearing loss, learning disabilitys, cognitive limitations, limited movement, speech disabilities, photosensitivity or a combination of these.

While this is a baseline for 508(C) compliance, that goes into a little more detail and has higher standards, generally speaking we want to follow WCAG 2.0 standards, and if the client is requesting 508(C) compliance we will use an auditing company to tell us what to fix.

There are four main areas this guideline covers, perceivable, operable, understandable, and robust.

  1. Perceivable
    • Provide text alternatives for any non-text content so that it can be changed into forms people need, such as large print, braille, speech, symbols or simpler language.
    • Provide alternatives for time-based media.
    • Create content that can be presented in different ways (for example simpler layout) without losing information or structure.
    • Make it easier for users to see and hear content including separating foreground from background.
  2. Operable
    • Make all functionality available from a keyboard.
    • Provide users enough time to read and use content.
    • Do not design content in a way that is known to cause seizures.
    • Provide ways to help users navigate, find contnet, and determine where they are.
  3. Understandable
    • Make content readable and understandable.
    • Make web pages appear and operate in predictable ways.
    • Help users avoid and correct mistakes.
  4. Robust
    • Maximize compatibility with current and future user agents, including assistive technologies.

For more information on these guidelines you can read the spec on www.w3.org.

Progressive Enhancement vs Graceful Degradation

As a team, and as a company, we try to take the approach of progressively enhancing our websites, as opposed to gracefully degrading the experience. While this may seem like it's saying the same thing in different ways, they are fundamentally different.

First, let's look at how we define each approach:

Progressive Enhancement: start by establishing our basic level. This is done by identifying what our base requirement of support is for the website, then building our base element to this specification. After that, we enhance the code by adding support for new features like transitions, transforms, transparent background elements, hover effects, etc. We should only be supporting features that are supported by each browser, accepting that there may be slight differences between browser experiences.

Graceful Degradation: start by building out the final result with all the bells and whistles. Then use plugins, polyfills, and hacks to make older browsers work the same.

We utilize modernizr.js to test whether a feature that we want to use is supported or not supported, then write our CSS accordingly. Modernizr.js will add a class to the html tag identifying if the desired feature you tested for is or isn't available for use. Then you can make different definitions based on that feature.

We do accept that each browser may vary in appearance slightly, however, the majority of the look and feel should be similar. Everything does need to function correctly, so a form has to submit or a wizard has to step you through each step of the process, but it doesn't have to animate between each step the same in every browser.

While we try to be strict on our approach, there are some exceptions. Especially where a couple small lines of JavaScript can make something work across all browsers.

Here are some things we want do, and how we want to do them:

Animations: all animations should be done using CSS wherever possible. You can trigger the animation by adding a class to the element or changing an attribute with JavaScript or jQuery, but the styles that are altered should be managed within CSS.

Transitions: same rules as animations.

SVG: use SVG for icons and images as much as possible. This may require the use of a third party tool like SnapSVG to do more sophisticated drawing and animations.

Text Shadows and Box Shadows: use CSS only to achieve this. If they don't exist in older browsers, that's an acceptable difference. It's not worth the additional markup or code and use of images to create such small effects.

Rounded Corners: use CSS border-radius for this feature. If an older browser doesn't support it, they will just get squared off corners. Again the extra weight of markup and images is not worth it to create rounded corners.

Video: using HTML 5 video for backgrounds or even self-hosting videos is a great approach. However, get a Flash player to fallback to using the HTML tags properly to work on older browsers. For the backgrounds, just load an image instead of the video.

Use Modernizr

Modernizr.js is an online utility that allows us to identify and detect if a browser supports a special feature. Outlined in the Progressive Enhancement vs. Graceful Degradation section, we said why we want to detect features over browsers and supply fallbacks for styling or features that may not be supported. We always want to try to enhance the experience over degrading, so keep that in mind when using detection.

When using this tool, you can create a version custom built for your website, or you can load all the detections. Generally, it's easier to add the detections as you build the website and keep updating the file. The great thing about the download you receive when building Modernizr on their website is there will be a URL that you can go to that will have all your previously selected elements saved. Keep that in the JS file so that someone coming in later can use that link to select the right items you've already setup for detection.

Version Control Standards

These standards apply across all languages and are mainly implemented for version control so that you only see the changes you make.

  1. Four Spaces, NO Tabs

    ALL of our coding teams need to follow this in whatever IDE you choose to use. Make sure the default for a tab is equal to inserting 4 spaces regardless of language.

  2. Don't pull from Server

    If you have your site setup in an FTP client, don't pull files from the Remote, always push your local files or let automated build servers do the build for you. Pulling the latest on the server may result in undesired changes being made.

  3. Only commit changes you made

    Review your commit and make sure the code you are committing is code that you actually altered. If there's something you didn't do, review it carefully before committing to Git.

Naming Conventions

  1. Camel Case IDs

    This will help distinguishing the difference between an ID and a class.

  2. Dash Class Names

    Multi-word class names should be separated using dashes, this helps distinguish them from IDs.

  3. Naming Classes

    Class names should be related to what the object is used for, starting with the generic item then adding on a more specific use case. For example, when creating multiple carousels for a site each carousel class should start with carousel- then have something more specific that you can target from there like carousel-hero and carousel-testimonial.

    Name the class for use, not location or color, as that could cause issues if the color or location were to change.

HTML Standards

Follow these standard practices when writing HTML:

  1. Write Semantic Meaningful Markup

    With HTML5 we have the addition of meaningful tags like the header, main, footer, article, section, aside, and nav tags. Use them, not only for making your content more accessible for users, and digestible by crawlers, but use also to apply CSS.

    We shouldn't see something like <header role="header" class="header">. That is a lot of redundancy and all that should be needed in this case is <header>. Style the main site header based on location to the body, or if working in DNN or other CMS's that add wrapping elements style based on it's relation to the wrapping element. Then if you have a hero on the page in your main article, make that a <header> and style it in corellation to the <article> or <main>.

    This will not only remove code from your HTML but will shrink your CSS substantially as well.

  2. Keep the Code Layer Separate from the Design Layer

    Remember to take the MVC approach of Model|View|Control when building even a static website. Your View layer should be separated from your Control layer. HTML should only be used for the control layer, while CSS will be in place for controlling look and feel.

CSS Standards

Follow these standard practices when writing CSS:

  1. Each Property On a Line

    Writing this way helps with version control so you can see exactly what changed within each commit. It will also help with resolving merge conflicts.

    Since most of our sites use LESS, they will be compiled into a single line and minified. So, readability in the file is more important than file size.

  2. Avoid The Use of IDs

    Avoiding ID's in CSS creates more re-usable code since ID's can only be used once on a page while classes can be used multiple times.

    The other issue when working with CMS sites is often ID's are controlled and created dynamically causing them to be unreliable if the component is deleted and re-added to a page.

JavaScript Standards

Follow these standard practices when writing JavaScript:

  1. Use jQuery

    As far as JavaScript libraries go, we've chosen jQuery as our standard library. While other libraries exist, this remains one of the most common libraries and unless a client requests us to use something different jQuery should be used on all R2i projects.

  2. Use Bootstrap

    As far as JavaScript frameworks go, we've chosen to utilize Bootstrap for our standard framework. While we don't load Bootstraps CSS file and have chosen to move away from the weight that it creates, we still want to utilize the JavaScript framework. Base CSS files have been created in the boilerplates to give us the styles needed to work with the Bootstrap JavaScript framework.

  3. Use .on() To Bind Events

    Instead of using .click(), .mouseenter(), .mouseout(), etc to bind a function, use the .on() and .off() to bind and unbind functions to the those events. This will make it easier to bind and unbind events, and is the jQuery standard even though those events still work for the time being.

    Do this for all events, except .ready() as it could cause issues.

  4. Be Consistent

    Be consistent with your use of punctuation in JavaScript, it doesn't matter if you use ' or ", $ or jQuery, just pick an approach and be consistent with that throughout the project.

  5. Use Plugins When Necessary.

    Feel free to utilize additional plugins as needed, but try not to use one if it's a simple 5-10 line function, and check out our standards for things before choosing a different plugin.

    We have standard plugins for Selects, Date Picker, Carousels, Tabs, Accordions, Modals, etc.

Folder Structure

For any project, download the boilerplate for the type of project you are working on. This will start you off with the correct folder structure, and include some base files to work from.

Here is a basic overview of the folder structure and naming conventions used within them, along with some basic guidelines:

The following example is made with a static site in mind, some folder uses may be different with different CMS's.

  • Site Root - this may vary with different CMS
    • resources
      • fonts - all font files used on the site should be located in this folder
      • images - all image files used on the site should be located in this folder
        • base - this folder should contain images used site wide
        • page name - create folders for each pages images
      • js
        • base - this folder should contain js files used site wide
        • component name - this folder should be created for each component and define javascript needed for specific components used throughout the site
      • less
        • base - this folder should contain less files used site wide
        • component name - this folder should be created for each component and define javascript needed for specific components used throughout the site
      • videos - all video files used on the site should be located in this folder
        • base - this folder should contain videos used site wide
        • page name - create folders for each pages videos
    • page folders
      • html files
    • html files

Desktop Operating Systems:

Windows: Microsoft actively supported versions of Windows, currently Windows 7, 8, 8.1, and 10.

Desktop Browsers:
Internet Explorer - 9 & 11
Edge - current release + 2 versions back
Firefox - current release + 2 versions back
Chrome - current release + 2 versions back
Safari - current release + 2 versions back (on Apple products only)

Mobile and Tablet Operating Systems:
iOS: current release + 1 version back
Android: current release
Windows Phone: current release

Mobile Browsers:
iOS: Chrome and Safari latest release only
Android: Chrome latest release only
Windows Phone: Edge/Internet Explorer current release only
